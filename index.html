kestrel

Kestrel is a cross-platform web server for ASP.NET Core based on libuv, a cross-platform asynchronous I/O library.

Microsoftdocs.microsoft.com/en-us/aspnet/core/fundamentals/servers/kestrel?tabs=aspnetcore2x

通过微软的定义：可知Kestrel 是一个跨平台的，异步IO的library。
Kestrel支持：Https, websocket, Unix Socket

Kestrel工作方式：

  1. without reverse proxy
    Internet <->(http) ASP.NET Core application ( Kestrel <-> (httpContext) code )

  2. with reverse proxy:
    Internet <-> (http) Reverse proxy (IIS/windows, Nginx/linux) <->(http) ASP.NET CORE application ( Kestrel <-> (httpContext) code )

Tip：
反向代理：安全（多一层代理），load balance （平衡webserver负载），静态资源的缓存等等；另外，由于代理转发，故可以让多个web应用共享同一port和IP，

Kestrel在ASP.NET CORE中的应用：

IWebHostBuilder 扩展方法：UseKestrel()
同时可以设置限制的配置选项：

  1. MaximunClientConnection 最多的客户端连接数 default is null
  2. MaximunRequestBodySize 最大的请求Body大小 default is 30000000bytes
  3. MinimunRequestBodyDataRate Kestrel 检查传输速度，如果传输速度太小 close the connection.

aspnetcore2.0_middleware

中间件(Middleware):

作用

中间件集成在asp.net pipeline，用来处理请求和响应：

  1. 决定将请求是否交由下一个请求管道处理;
  2. 根据需求设计中间件的执行顺序
    request delegate用来构建请求管道（pipeline），pipeline本质上.net上也就是delegate。

与传统module，handler的区别

middleware 和传统的asp.net handler和module的区别：

  1. asp.net module需要配置 web.config, Global.asax.cs等很多文件，而middleware只需写csharp代码
  2. asp.net module的顺序需要借助asp.net 生命周期，而middleware则更加灵活，可以通过csharp来替换
  3. middleware request和reponse的顺序是相反的, 而module 是相同的

使用

  1. Run: teminal middleware
  2. Use: adding middleware
  3. map: map 映射

中间件的表现形式：匿名方法，可重用的类

次序

中间件的次序定义在configure，中间件影响着软件的安全，性能和可扩展。

aspnetcore2.0_websocket

WebSocket是HTML5下一种新的协议。它实现了浏览器与服务器全双工通信，能更好的节省服务器资源和带宽并达到实时通讯的目的。它与HTTP一样通过已建立的TCP连接来传输数据，但是它和HTTP最大不同是：

  • WebSocket是一种双向通信协议。在建立连接后，WebSocket服务器端和客户端都能主动向对方发送或接收数据，就像Socket一样；
  • WebSocket需要像TCP一样，先建立连接，连接成功后才能相互通信。

特点：

  • 不同于轮询，websocket建立连接后，客户端与服务器是完全平等的，可以互相请求。
  • websocket不需要HTTP Header来验证身份，websocket更加轻量，传输速度也更加快。
  • 没有同源限制
  • 与Http协议保持良好的兼容性，默认端口也是80或者443
  • 基于HTTP请求头进行握手，会有sec-websocket-key计算，目的是验证服务端是否具有建立websocket的能力

websocket通常应用于浏览器和服务器的通讯，但是也可以应用于客户端和服务器的通讯。

Action Filter

Asp.net core 2.0 中根据内嵌的不同Filter运行在请求管道（request pipeline）的不同阶段。主要用途有：

  1. authorization 验证权限
  2. 根据请求的文件不同，来缩短Request pipeline。比如：有些静态资源的请求
  3. 对于请求接口参数的Validation，来实现解耦
    等等

filter的执行顺序：

  • Authorization Filter 用于确定当前用户的请求是否合法。

  • Resource Filter 是授权之后第一个用来处理请求的过滤器，也是最后一个接触到请求的过滤器（因为之后就会离开过滤器管道）。在性能方面，资源过滤器在实现缓存或短路过滤器管道尤其有用。

  • Action Filter 包装了对单个 action 方法的调用，可以将参数传递给 action 并从中获得 action result。

  • Resource Filter 为 MVC 应用程序未处理异常应用全局策略。

  • Result Filter 包装了单个 action result 的执行，当且仅当 action 方法成功执行完毕后方才运行。它们是理想的围绕视图执行或格式处理的逻辑（所在之处）。
